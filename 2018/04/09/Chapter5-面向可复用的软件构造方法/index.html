Programming for/with reuse

5.1.1 About Reuse

可复用性：不针对应用，而是针对多个引用场景。

Software reuse

Two perspectives of software reuse
programming for reuse(面向复用编程，开发出可复用的软件->抽取共性，识别差异，设计继承树) programming with reuse(基于复用的编程，利用已有的可复用的软件搭建应用系统->搜索，选择适配)
开发成本高于一般软件的成本：要有足够高的适应性；性能差些：针对更普适场景，缺少足够的针对性 可复用软件库，对其进行有效的管理；往往无法拿来就用，需要适配

Why reuse？

Why reuse?
Reuse is **cost-effective** and with **timeliness**(降低成本和开发时间) Reuse produces **reliable** software(经过充分测试，**可靠、稳定**) Reuse yields **standardization**(**标准化**，在不同应用中保持一致)

Measure resuability
复用的机会有多频繁？
复用的场合有多少？
复用的代价有多大？（搜索、获取；适配，扩展；实例化；与软件其他部分的互连）

A software asset with high reusability should:
– Brief (small size) and Simple (low complexity) 小、简单
– Portable and Standard Compliance 与标准兼容
– Adaptable and Flexible 灵活可变
– Extensibility 可扩展
– Generic and Parameterization 泛型、参数化
– Modularity 模块化
– Localization of volatile (changeable) design assumptions 变化的局部性
– Stability under changing requirements 稳定
– Rich documentation 丰富的文档和帮助

5.1.2 Levels and morphology of reusable components

Type of Code Reuse（代码复用的类型）

White box reuse(白盒复用) Black box reuse(黑盒复用)
源代码可见，可修改和扩展（复制已有代码当正在开发的系统，进行修改，可定制化程度高，对其修改增加了软件的复杂度，且需要对其内部充分的了解） 源代码不可见，不能修改（只能通过API接口来使用，无法修改代码，简单、清晰，适应性差）

Reuse Level（复用级别）

Source code level reuse（源代码级别复用） Module-level reuse（模块级别复用） Library-level（库函数级别） System-level reuse（系统级别）
methods, statements class/abstract class/interface API/package(.jar) framework(框架)
Copy/paste；Exampel code search: ①grepcode.com ②github.com/search ③searchcode.com Approaches of reusing a class: inheritance(继承, override), delegation(委托，一个对象将某些功能传递给另一个对象) Libraries: A set of classes and methods(APIs) that provide reusable functionality.开发者构造可运行软件实体，其中涉及到对可复用库的调用 框架：一组具体类、抽象类、及其之间的连接关系；开发者根据framework的规约，填充自己的代码进去，形成完整系统；**Framework作为主程序**加以执行，执行过程调用开发者所写的程序

Details For Frameworks（关于框架）

framework
Whitebox frameworks（白盒框架） Blackbox frameworks（黑盒框架）
通过继承（inheritance）和动态绑定（dynamic binding）实现的可扩展性；通过继承（subclassing）框架基类并重写（overriding）预定义的hook(钩子)方法来扩展现有功能；通常使用诸如模板方法模式的设计模式来覆盖钩子方法 通过为可插入框架的组件定义接口来实现可扩展性； 通过定义符合特定接口的组件来重用现有功能； 这些组件通过**委托**（delegation）与框架集成
通过**subclassing**(继承)、**override**实现定制功能；允许扩展每个非私有方法；需要了解超类的实现；每次只有一个扩展；一起编译；通常称为开发框架 通过**plugin interface**(**delegation**、**composition**)实现；允许扩展在接口中显示的功能；只需要了解界面；多个插件；通常提供更多的模块化；单独部署称为可能（.jar，.dll，...）；通常所谓的最终用户框架、平台
程序执行的主体是开发者实现的**子类** 程序执行的主体是**framework**
Domain Engineering（领域工程）：①定义域（domain）②分解和实施通用部件为框架③为可变部分提供插件接口和回调机制（callback）④获取大量反馈，并进行迭代
运行框架：①通过自身运行（eg. Eclipse）②通过继承运行（eg. Swing，JUnit，MapReduce，Servlets）③加载插件（Client写main函数，创造插件并传递给框架（白盒）；框架写main函数，client将插件名字作为命令或环境变量传递（黑盒））
5.1.3 External observations of reusability
Type Variation(类型可变) Routine Grouping(功能分组) Implementation Variation(实现可变) Representation Independence(表示独立) Factoring Out Common Behaviors(共性抽取)
泛型：适应不同的类型，且满足LSP 提供完备的细粒度操作，保证功能的完整性，不同场景下复用不同的操作及其组合 ADT有多种不同的实现，提供不同的representation和abstract function，但具有同样的specification(pre-condition,post-condition,invariants)，从而可以适应不同的应用场景 内部实现可能会经常变化，但客户端不应受到影响，表示独立性、信息隐藏 将共同的行为（共性）抽象出来，形成可复用实体

5.2.1.4 Behavioral subtyping and Liskov Substitution Priciple(LSP)(行为子类型与Liskov替换原则)

子类型多态：客户端可用统一的方式处理不同类型的对象(static type checking)

子类型在Java编译器中执行的规则：
①子类型可以增加方法，但不可删
②子类型需要实现抽象类型中的所有未实现方法
③子类型中重写的方法必须有相同或子类型的返回值
④子类型中重写的方法必须使用同样类型的参数
⑤子类型中重写的方法不能抛出额外的异常
特殊的行为：
⑥更强的不变量；
⑦更弱的前置条件；
⑧更强的后置条件；

Liskov Substitution Priciple(LSP): Let q(x) be a property provable about objects x of type T, then q(y) should be provable for objects y of type S where S is a subtype of T.
（对于T类型的对象x有q(x)成立，且S是T的子类型，则对于S类型的对象y有q(y)成立）

LSP是子类型关系的一个特殊定义，称为强行为子类型。
在编程语言中遵循以下规则：
①前置条件不能强化
②后置条件不能弱化
③不变量要保持
④子类型方法参数：逆变（反协变，越来越抽象）
⑤子类型方法的返回值：协变（越来越具体）
⑥异常类型：协变（越来越具体）

Covariance(协变) Contravariance(反协变、逆变) 两种特别的LSP Wildcards（通配符）
父类型->子类型：越来越具体specification； 返回值类型：不变或变得更具体； 异常的类型：也是如此 父类型->子类型：越来越具体specific； 参数类型：要相反的变化，要不变或越来越抽象 数组中的LSP 泛型中的LSP 解决泛型中的LSP问题
Example: ①返回值类型，Object a(){...} -> String a(){...} ②异常类型，throws Throwable -> throws IOException Example: ①参数类型，void c(String s){...} -> void c(Object s){...} Arrays are covariant.(定义抽象类型可以放入具体类型，父类型可以放入子类型) ArrayList<Sting> 是List< Sting> 子类型； List<Sting> 不是 List<Object> 子类型； 这个过程称为类型擦除（泛型不是协变的） List<? extends T> (Upper Bounded Wildcards，T的子类型) ； List<Number> 是 List<?> 子类型； List<Number> 是 List<? extends Object> 的子类型（Number是Object/Number等子类的子类型型）； List<Object> 是 List<? super String/Number/Object> 的子类型（Object是String/Object等任意类型父类的子类型）；
关于上述问题代码分析： ①子类型中参数类型协变；需要注意，如下情况不是子类型，因此是正确的 ②数组的LSP，需要注意子类型对象赋值和子类型引用赋值，前者正确，后者错误 ③泛型中的LSP，泛型不是协变的 ④通配符的应用，使用通配符的泛型的LSP
5.2.1.5 Composition and delegation(组合和委托)

Delegation & CRP原则

Delegation(委托/委派) Composition Reuse Principle（CRP，复合重用原则，Or Composite over inheritance Principle）
一个对象请求另一个对象的功能(Client调用Recevier，Recevier委派给Delegate；委派是复用的一种常见形式)；如果子类只需要复用父类中的一小部分方法，可以不需要使用继承，而是通过委派机制来实现 类应该通过它们的组合（通过包含实现所需功能的其他类的实例）实现多态行为和代码重用，而不是从基类或父类继承；优先使用对象组合，而不是类继承
establish delegation between two objects； A a = new A()； B b = new B(a)(Class B内部存在一个delegation link -> private A a，如果B要使用A的foo()功能，则可以直接调用a.foo()实现) 使用接口定义不同侧面的行为；接口之间通过extends（继承）实现行为的扩展（一个接口可以继承多个接口- > 组合）；类implements（实现）接口（类的内部使用委派）

Types of delegation（三种委派方式）

Types of delegation
Use（A use B） Association（A has B） Composition / aggregation（A owns N）
Dependency（依赖，临时性的delegation） Association（关联，永久性的delegation） Composition（组合，更强的delegation） Aggregation（聚合）
直接调用其他对象的方法（uses_a） 这种关系是结构性的，因为它指定一种类型的对象与另一种类型的对象相连，并不代表行为（has_a） 将简单对象或数据类型组合成更复杂的对象；该类有另一个作为属性/实例变量（内部创建实例对象，is_part_of） 对象存在于另一个之外，在外部创建，所以它作为参数传递给构造者（has_a）

5.2.2 Designing system-level reusable libraries and frameworks（系统层面复用）

之所以library和framework被称为系统层面的复用，是因为它们不仅定义了1个可复用的接口/类，而是将某个完整系统中的所有可复用的接口/类都实现出来。并且定义了这些类之间的交互关系、调用关系，从而形成了系统整体的“架构”。

5.2.2.1 API design

(1) API should do one thing and do it well（努力做好一件事情；Good. Font，PrivateKey，ThreadFactory；Bad. BindingIteratorImplBase）
(2) API should be as small as possible but no smaller（越小越好）
(3) Implementation should not impact API（信息隐藏，用户不应该影响API）
(4) Documentation matters（文档遵循Chapter4的可读性）
(5) Consider performance consequences（性能良好）
(6) API must coexist peacefully with platform（友好性）
(7) Class design（最小化可变性；继承遵循LSP）
(8) Method design（编译时最好 - 静态类型、泛型）
